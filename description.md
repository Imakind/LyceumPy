# Описание проекта: Aqbobek Lyceum Portal

Проект представляет собой прототип единого образовательного портала для лицея, включающий элементы расписания, дневника успеваемости, новостной ленты и ИИ-наставника (Alaman).

## 1. Архитектура и Структура
В проекте наблюдается наличие **нескольких параллельных слоев приложения**, которые решают похожие задачи, но реализованы по-разному.
- **Backend (FastAPI)**: Находится в файле `main.py`. Реализует REST API (Kiosk Mode, Analytics, Mock BilimClass) с подключением к PostgreSQL через SQLAlchemy.
- **Frontend 1 (Streamlit)**: Находится в файле `app.py`. Это полноценное монолитное веб-приложение на Python, которое имеет собственную систему авторизации, базу данных на JSON-файлах и встроенный UI. 
- **Frontend 2 (Next.js)**: Находится в папке `frontend`. Реализован на React и TailwindCSS. Представляет собой современный дашборд-прототип с графиками Recharts.

Docker Compose файл (`docker-compose.yml`) связывает базу данных Postgres, бэкенд на FastAPI и фронтенд на Next.js. Однако приложение на Streamlit запускается отдельно или предполагается как альтернативный вариант.

## 2. Логика и Функционал
Бизнес-логика приложения во многом завязана на имитации (Mock) данных:
- **Ролевая система**: Поддерживаются роли `student`, `teacher`, `parent`, `admin`. Для каждой роли отображается свой интерфейс.
- **Дневник и Успеваемость**: Учителя могут ставить оценки. Система анализирует средний балл и строит тренды успеваемости.
- **ИИ-Наставник (Alaman)**: Интеллектуальный помощник (реализован в `bilim_engine.py`). Имеет предиктивную модель, вычисляющую вероятность успешной сдачи СОЧ на основе последних оценок. Также может подключаться к OpenAI API, если задан ключ, иначе работает по эвристическим алгоритмам.
- **Генерация расписания**: Система может генерировать расписание без конфликтов (`generate_schedule_conflict_free`) и обрабатывать ситуации болезни учителя (замена на свободного преподавателя).
- **Система аномалий**: Отслеживание резкого падения успеваемости (разница > 20 баллов) и формирование отчетов для директора.

## 3. База Данных и Управление Состоянием
В проекте присутствует **двойная логика хранения данных**:
1. **Реляционная БД (PostgreSQL)**: Описана в `models.py` и `database.py`. Содержит таблицы `users`, `students`, `teachers`, `classes`, `subjects`, `grades`, `schedule`, `rooms`. Используется в FastAPI.
2. **Локальные JSON-файлы (State)**: Используются в Streamlit приложении (`users_db.json`, `news_db.json`, `messages_db.json`, `bilim_data.json`). Streamlit активно использует `st.session_state` для хранения текущих данных сессии (авторизация, лайки, текущий ученик) и синхронизирует их с JSON для персистентности.

## 4. Дизайн и Стилизация (UI/UX)
- **Streamlit (`app.py`)**: Используется встроенный CSS (`crm_css()`), который переопределяет стандартный вид Streamlit. 
  - Цветовая палитра: Primary `#0c1e3d` (темно-синий), Accent `#3d8bfd`.
  - Используются скругленные карточки (`border-radius: 20px`), градиентные фоны, скрытие стандартного сайдбара Streamlit для учеников (`student_hide_sidebar_css`), и плавающая кнопка (FAB) для вызова чата с ИИ.
- **Next.js (`frontend/`)**: Использует TailwindCSS.
  - Ультрасовременный стиль с эффектами `glassmorphism` (backdrop-blur), мягкими тенями, большими скруглениями (`rounded-[2rem]`) и градиентами (indigo, rose). 
  - Реализованы красивые линейные графики через `recharts`.

## 5. Баги, Проблемы и Технический Долг
1. **Рассинхронизация архитектуры (Главный баг)**: Streamlit-приложение (`app.py`) и FastAPI-бэкенд (`main.py`) не связаны между собой. Streamlit работает с JSON-файлами, а FastAPI с PostgreSQL. 
2. **Изолированный Frontend**: Next.js-интерфейс (`page.tsx`) содержит "захардкоженные" данные в компонентах (например, графики и список группы риска), и не выполняет реальных fetch-запросов к FastAPI бэкенду (кроме импортов `calculateRating`, `getAIPrediction`, которые не привязаны к основному стейту).
3. **Заглушки (Mocks)**: 
   - В `scheduler.py` и `ai_service.py` методы статические и возвращают константные строки, не взаимодействуя с передаваемой сессией базы данных.
   - Нет реальной валидации JWT или токенов в FastAPI, авторизация в Streamlit держится на простых проверках пароля из JSON.
4. **Конкурентный доступ в Streamlit**: Чтение/запись в файлы `.json` (`save_users`, `save_news`) реализовано без блокировок. При одновременной работе нескольких пользователей могут возникнуть конфликты записи (Race Conditions) и повреждение JSON-файлов.

## Итог
Проект представляет собой качественный и визуально проработанный прототип (MVP) для демонстрации идеи "Умного Лицея" на хакатоне. Бизнес-эвристики (вычисление трендов, графов знаний, предиктивная аналитика) прописаны хорошо. Однако для перевода в production (или полноценный релиз) требуется объединить разрозненные части: перевести весь UI на Next.js (или полностью на Streamlit), а бизнес-логику из `bilim_engine.py` и json-хранилищ перенести в REST API на FastAPI и PostgreSQL.
